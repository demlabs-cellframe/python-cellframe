# Недостатки и проблемы модуля chain (обёртки)

## Критические
- Мутация блоков (`dap_chain_block_*_add`) может приводить к инвалидным указателям: SDK принимает `dap_chain_block_t **` и может realloc, но капсула в Python не обновляется. Это риск UAF/краша при дальнейшем использовании.

## Высокий приоритет
- Некорректное освобождение `l_atom_map`: в `dap_chain_atom_save_py` и `dap_chain_cell_file_append_py` указатель может ссылаться на mmap‑область, но освобождается через `DAP_DELETE`, что может приводить к крахам/повреждению памяти.
- Утечки в жизненном цикле callbacks: колбэки/`user_data` инкрементируются дважды (в обёртке и в реестре), при этом нет механизма снятия/очистки контекстов, что ведёт к утечкам объектов Python и `python_chain_callback_ctx_t`.

## Средний приоритет
- В режиме без настроенного хранилища callback вызывается под `chain->rwlock`. Если Python‑колбэк реэнтрантно вызывает chain‑API, это может привести к дедлоку.
- `dap_chain_cell_capture_by_id_py` возвращает капсулу с удержанным rwlock; если пользователь не вызовет `dap_chain_cell_remit_py`, возможна блокировка других потоков.

## Низкий приоритет / техдолг
- Отсутствуют деструкторы/явные освобождения для ряда объектов, возвращаемых в Python (policy, chain packet, block, mempool datum, list). Это создаёт утечки при долгоживущих процессах.
- `dap_chain_srv_purge_py` освобождает список без корректного освобождения элементов (UID в списке создаются как `DAP_DUP`).

## Дополнительные замечания
- Докстроки и соглашения о владении ресурсами (ownership) не зафиксированы, из‑за чего поведение по освобождению/времени жизни объектов неочевидно для пользователя.
